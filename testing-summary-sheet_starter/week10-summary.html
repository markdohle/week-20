Create-React-App

Create-React-App is a command you can run via the command line that creates an entirely configured React application for you, complete with Babel setup, recommended folder structure, and some skeleton files, like a skeleton App.js. To run this command, run this command in your command line: 

npx create-react-app

After some time downloading, you'll have an entire React application set up so that you can start editing the files for your own purposes right away!

GraphQL

GraphQL is an API query framework that allows you to specify exactly what data you want from an API. This helps with load time by eliminating excess data, and allows for more flexibility in the client/server communication stream.

For example, if you had an API that had data on gyms in the area (name, hours, amenities, membership costs, etc.) and you only wanted to get the name and ID of all the gyms for your app, if you went through a traditional API query method, you'd have to wait for a long time for all the data and have a lot of extra data to sort through in your app.

A GraphQL query for just the names and IDs of the gyms looks like this:

{gyms {id, name}}

To see an example in real time, review Dr. William's video, "GraphQL demo".

Testing and TDD

Testing is such a vast field in engineering that there are entirely separate jobs dedicated to it, such as those of Quality Assurance Engineers and Test Engineers.

Test driven development is a recent development in the software engineering industry. The process of engineering in this manner involves writing a test for the functionality you want your code to have, making the test fail, and then writing code in an iterative process to make the test pass. This allows for far fewer bugs and a much faster development lifecycle overall.

For example, if you want to find out if your application has an H1 tag that contains the text, "ToDo", the test would look like this: 

test('ToDo', () => {
  const root = document.createElement('div');
  ReactDOM.render(<App/>, root);

  expect(root.querySelector("h1").textContent.toBe("ToDo");
});

Mocks

When you're testing your code, oftentimes you’ll need to test your components which have a dependency on some other service that has unpredictable results, or takes a long time to receive a response from. What you’ll do in those situations is add a “mock”, or fake dependency, so that you can focus on testing only what you're building, not the dependency’s functionality.

Here's an example of some React code that displays this is my component to the screen: 

import React from ‘react’;

export default function myComponent() { 
  return (
   <div>this is my component</div>
  );
}

Now, if you were testing that code, normally you'd test for something like that exact phrase being displayed as expected. But what if you wanted to instead "mock" that component in your test file, and make it say something else entirely?

That'd look like this:

import React from ‘react’;
import { render } from ‘@testing-library/react’;
import App from ‘./App’;

//mock your component here 
jest.mock (‘./MyComponent’, () => () => (<div>Hello, World!</div>));

test (‘mocking test’, () => {
  //test contents here 
});

As you can see, you're replacing the "this is my component" statement in your original component with some new text, and testing if the new text you have put in is displayed instead. This type of functionality becomes important when you have data that you don't necessarily want to test, but you need for some other section of your code to run. Instead of using data that's large or unpredictable, you can "mock" that data, and focus on the code you really want to test.
